# 1.缓存

redis在业务里的作用就是起到缓存的作用 
缓存的定义：缓存是一种具有高效读写能力的区域

由于redis是基于内存的数据库，因此在想redis中插入数据是就涉及到了内存的问题，即缓存更新策略

# 2.缓存更新策略
策略一般为三类：
- 使用redis自带的内存淘汰策略
- 设置TTL，过期时间
- 主动更新：当修改数据库数据的时候同时去更新缓存
## 如何选择？
- 对数据一致性要求低:采用redis自带的内存淘汰策略即可，无需额外编码
- 对数据一致性要求高：主动更新，并辅助以过期时间

## 主动更新？
如何主动更新？即同时修改数据库的数据以及缓存
建议采用删除缓存，而非进行缓存修改。
### 先更新缓存还是先更新数据库？
这是一个线程问题，无论先更新哪个都有概率会发生数据不一致。
但是缓存是在基于内存的，因此修改速度快，所以更推荐先更新数据库，然后修改缓存。

由于设置了过期时间，会存在数据会在某一段时间内过期的现象，因此又会出现不同的问题
# 3.缓存穿透
问题描述：用户查询的数据在缓存中和数据库中都不存在，该请求就会全部打到数据库上，对数据库的压力剧增

解决方案：
- 缓存空对象：当查询缓存未命中后，查询数据库也没有查到数据，然后在缓存中加入""空字符串，代表没有数据，下次再查询没有的数据时就会直接缓存命中，返回null代表数据库中没有该数据
- 布隆过滤：一种基于哈希的算法，没学过，待完成

缓存过期还会存在在同一时刻大量的数据过期或者redis服务直接崩溃这种极端现象，导致同一时刻大量的请求打到数据库，该现象成为缓存雪崩
# 4.缓存雪崩
缓存雪崩还是比较好解决的，问题在于同一时刻大量数据过期，所以可以在设置过期时间的时候加上一个随机值，让过期时间落在一个时间段内分批次的过期，这样就减小的数据库的压力

如果redis服务宕机了。就存在redis集群的问题

缓存雪崩是同一时刻内大量数据过期导致的问题，同时实际开发中还存在一种现象，是少量的key过期也会导致数据库压力剧增，该问题为缓存击穿

# 5.缓存击穿
缓存击穿：也成为"热点key问题"，即某个key会被高并发的访问或者其数据重构时间久导致的问题。一旦热点key过期，高并发的访问会直接到达数据库。

解决方法：当数据重构的期间，不要让高并发的请求到达数据库

优化方法：
- 互斥锁：当一个请求到达缓存的时候，没有查询到数据，先请求该对象的互斥锁，如果请求成功，就去重构对象并写入缓存中，如果请求失败，就进行等待，随后再次请求该数据，直到获取锁的线程重构成功对象并写入缓存，随后返回数据

  所有进程都在等待获得锁的进程完成对象的重构，效率低下。
- 逻辑过期：说到底会出现缓存穿透，缓存雪崩，缓存击穿这些问题全部都是因为数据会过期，会从redis里删除，因此设置数据永不过期，我们不让数据真正的过期，而是让他逻辑上过期，实则redis里依旧存在该数据。当一个请求到达缓存的时候，查询到数据并发现他没有过期的时候，直接返回数据即可。查询不到数据，线程会先尝试获取互斥锁，获取成功后他会指派给其他进程让该进程去重构对象，然后直接返回旧的结果。其他请求在到达缓存查询数据后，获取不到互斥锁，直接返回旧的结果，知道对象重构完成，这时请求来之后才会返回新的结果

    可以看出该方法会有数据不一致的问题。别问有新的数据了旧的怎么办，旧的放转转回收了。

  
如何选择这两种方法呢？

取决于业务的要求，看你追求高性能还是数据的一致性
